# Руководство по участию в проекте (Contributing)

Мы рады вашему вкладу! Чтобы сделать процесс разработки прозрачным и эффективным, пожалуйста, следуйте этим правилам.

## Рабочий процесс

Весь процесс разработки описан в документе [DEVELOPMENT_WORKFLOW.md](DEVELOPMENT_WORKFLOW.md). Ключевые моменты:

1.  Создавайте ветку `feature/...` или `bugfix/...` от `develop`.
2.  После завершения работы создайте Pull Request в ветку `develop`.
3.  Убедитесь, что ваш PR проходит все автоматические проверки (CI).
4.  Ваш PR должен быть одобрен как минимум одним членом команды.

## Стиль кода (Code Style)

- **JavaScript/TypeScript:** Мы используем [Prettier](https://prettier.io/) для автоматического форматирования кода. Настройки уже включены в проект. Перед коммитом убедитесь, что код отформатирован.
- **React:** Мы следуем официальным [React Hooks rules](https://reactjs.org/docs/hooks-rules.html). ESLint автоматически проверяет их соблюдение.
- **Именование:**
  - Компоненты: `PascalCase` (e.g., `BottomNavigation`).
  - Переменные и функции: `camelCase` (e.g., `telegramUser`).
  - Типы и интерфейсы: `PascalCase` (e.g., `TelegramUser`).

## Комментарии

- Используйте `// TODO:` для обозначения функционала, который нужно будет добавить.
- Используйте `// FIXME:` для обозначения проблем, которые нужно исправить.

Наша система CI автоматически создаст задачу (Issue) из этих комментариев.

## Сообщения коммитов

Мы придерживаемся стиля [Conventional Commits](https://www.conventionalcommits.org/). Это помогает нам автоматически генерировать список изменений (changelog).

**Формат:** `<type>(<scope>): <subject>`

- **`feat`:** Новая функциональность.
- **`fix`:** Исправление ошибки.
- **`docs`:** Изменения в документации.
- **`style`:** Изменения, не влияющие на логику (форматирование).
- **`refactor`:** Рефакторинг кода без изменения поведения.
- **`test`:** Добавление или исправление тестов.
- **`ci`:** Изменения в CI/CD.

**Пример:** `feat(auth): add support for Apple ID login`
